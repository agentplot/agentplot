### Requirement: subcog client role uses mkClientTooling
The subcog client role SHALL be generated by `mkClientTooling` with capabilities: `skills = [ ./skills/cli/SKILL.md ]`, `cli = { package, wrapperName, envVars }`, `mcp = { type = "http"; urlTemplate; extraConfig }`, `secret = [ db-password (shared), jwt-token (shared) ]`. Extra client options: `domain`, `namespace`.

#### Scenario: CLI wrapper exports SUBCOG_DOMAIN
- **WHEN** a subcog client is configured with `domain = "subcog.swancloud.net"`
- **THEN** the generated CLI wrapper `subcog-<name>` SHALL export `SUBCOG_DOMAIN=subcog.swancloud.net`

#### Scenario: CLI wrapper exports storage connection
- **WHEN** a subcog client is configured with `domain = "subcog.swancloud.net"`
- **THEN** the generated CLI wrapper SHALL export `SUBCOG_STORAGE_BACKEND=postgresql` and `SUBCOG_STORAGE_CONNECTION_STRING` pointing to `postgresql://subcog:<password>@subcog.swancloud.net/subcog` with the password read from the shared `subcog-db-password` vars generator

#### Scenario: MCP config includes JWT tokenFile
- **WHEN** a subcog client is configured
- **THEN** the MCP config SHALL include `tokenFile` pointing to the shared `subcog-jwt-token` vars generator output

#### Scenario: subcog client with namespace isolation
- **WHEN** two subcog clients are configured with different namespaces
- **THEN** each SHALL get independent MCP entries that can target different namespace scopes

#### Scenario: shared secrets reference server generators
- **WHEN** a subcog client is configured
- **THEN** the `db-password` secret SHALL reference the server's `subcog-db-password` vars generator with `mode = "shared"`
- **AND** the `jwt-token` secret SHALL reference the server's `subcog-jwt-token` vars generator with `mode = "shared"`
- **AND** no new vars generators SHALL be created for these secrets on the client
